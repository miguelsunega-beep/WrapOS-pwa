# WrapOS — Contexto do Projeto

## Stack
React 18 + TypeScript 5.2 + Vite 5 + Tailwind + Framer Motion + Recharts + Sonner + lucide-react
Testes: Playwright (e2e/) — 6 specs cobrindo fluxo principal de OS, OS a receber, cancelamento, exclusão de cliente, estoque crítico. A suíte roda atrás do login do Supabase Auth: `e2e/global-setup.ts` loga uma vez com um usuário de teste já confirmado (credenciais em `.env.e2e.local`, nunca commitado) e salva a sessão em `e2e/.auth/user.json` (gitignored), reaproveitada por todos os specs via `storageState` no `playwright.config.ts`. A conta de teste é **dedicada só para e2e** (`teste-e2e@wrapos.local`), não uma conta pessoal/de uso real — já tem linha correspondente em `usuarios`, vinculada a uma loja de teste separada (`lojaId = e2e00000-0000-4000-8000-000000000001`, "Loja Teste E2E"), seguindo o processo em "Como adicionar um novo usuário/funcionário" abaixo. Isso evita misturar dado de teste com dado real e evita conflito se alguém estiver logado com a própria conta em outro navegador. Para rodar localmente, crie `.env.e2e.local` com `E2E_TEST_EMAIL` e `E2E_TEST_PASSWORD` dessa conta de teste (peça as credenciais, não use sua própria conta).

## Padrão de arquitetura (importante)
- **Toda lógica de negócio fica em hooks**, nunca na página (`.tsx` de página só renderiza, usa o hook correspondente). Página overgrown → extrair pra hook, não reescrever do zero.
- **Lógica de status/regra compartilhada entre telas vai em src/lib/** como função pura (ex: `getStatusEfetivo` em agendamentoStatus.ts, mapeamento de etapa em patioEtapa.ts) — nunca duplicar essa lógica dentro de um hook.
- Persistência: `usePersistedState` → localStorage com namespace por perfil (dentro de AppContext.tsx). O "perfil" hoje é sempre `usuario.lojaId` (ver "Perfil local = loja do usuário logado" abaixo) — não existe mais tela de seleção/criação manual de perfil.

## Arquitetura de Dados
- **Persistência ATUAL**: mista. `clientes` e `veiculos` já vivem no Supabase (ver "Migração de entidades pro Supabase" abaixo); as outras 9 entidades continuam em localStorage via `usePersistedState` em `AppContext.tsx`, namespace por perfil. Migração incremental, entidade por entidade — não é um corte único.
- **Banco**: Supabase (Postgres gerenciado), projeto "wrapos pwa latest version", região sa-east-1. Schema aplicado manualmente via SQL Editor do Supabase (não via migration automatizada).
- **Schema Prisma completo**: `prisma/schema.prisma` cobre hoje as 13 entidades — as 5 originais (Loja, Usuario, Cliente, Veiculo, OrdemServico) mais as 8 adicionadas depois (Agendamento, Instalador, LancamentoFinanceiro, Produto, Garantia, Meta, Configuracoes, Servico), espelhando os campos de `src/types/index.ts`. Todo model de entidade (exceto `Loja`) tem `lojaId: String` + relação/`@@index` para `Loja`, seguindo o padrão multi-tenant já existente. `Configuracoes` é a exceção: `lojaId` é `@unique` (relação 1:1, uma config por loja). **`Cliente` e `Veiculo` são exceção** (desde 2026-07-02 e 2026-07-06 respectivamente): chave primária composta `@@id([lojaId, id])` em vez de `id @id` sozinho — ver "Migração de entidades pro Supabase" abaixo pro motivo (colisão de id entre lojas) e o padrão a replicar nas próximas entidades migradas. Campos que apontam pra outra entidade "de negócio" (ex: `instaladorId` em `Agendamento`/`OrdemServico`, `osId` em `Garantia`/`LancamentoFinanceiro`, `servicoId` fora do próprio dono, `clienteId`/`veiculoId` em `Agendamento`/`Garantia`) ficam como `String`/`String?` simples, **sem relação Prisma** — não há FK entre esses models, só entre cada entidade e `Loja`, com uma exceção que agora cobre duas relações: `Veiculo.cliente` **e** `OrdemServico.cliente`/`OrdemServico.veiculo` **têm** FK composta (`(lojaId, clienteId) → Cliente(lojaId, id)` e `(lojaId, veiculoId) → Veiculo(lojaId, id)`), criadas junto com a chave composta de cada entidade referenciada (não dá pra ter uma FK simples apontando pra uma PK composta). `schema.sql` (raiz) é sempre a regeneração completa via `npx prisma migrate diff --from-empty --to-schema prisma/schema.prisma --script` — não editar à mão, sempre regenerar depois de mudar o schema.
- **Auth**: Supabase Auth (não Firebase). Implementado no frontend: `src/hooks/useAuth.ts` (signIn/signUp/signOut + estado de sessão, e busca a linha correspondente do usuário logado na tabela `usuarios` por `authUserId`) e `src/components/ProtectedRoute.tsx` (gate em torno de todo `App.tsx`, com três estados: `LoginPage` quando não há sessão, `ContaNaoVinculada` quando há sessão válida mas nenhuma linha em `usuarios` foi encontrada pra esse `authUserId`, e as páginas normais quando ambos existem). **Cadastro público está desabilitado**: `LoginPage.tsx` só expõe login (email/senha) e "Esqueci minha senha" (`supabase.auth.resetPasswordForEmail`). `signUp` continua exportado em `useAuth.ts` só para uso manual/scripts — novas contas são criadas por convite, não pela UI (ver "Como adicionar um novo usuário/funcionário" abaixo).
- **Prisma**: mantido **apenas** como schema-as-code (`prisma/schema.prisma`) para gerar SQL via `npx prisma migrate diff` — não há `PrismaClient` rodando no app. **Nunca rodar `npx prisma db push` ou `npx prisma migrate dev`** — a rede bloqueia conexão direta nas portas 5432/6543. `prisma migrate diff` entre dois schemas locais (`--from-schema`/`--to-schema`, ambos arquivos) não precisa de conexão com o banco e funciona normalmente.
- **Para alterar o banco**: editar `prisma/schema.prisma`, gerar o SQL incremental comparando a versão anterior do schema (salva antes de editar) com a nova via `npx prisma migrate diff --from-schema <schema-antigo.prisma> --to-schema prisma/schema.prisma --script`, salvar o resultado em `supabase/migrations/NNN_descricao.sql` (numeração sequencial — hoje até `004_veiculos_chave_composta.sql`), regenerar `schema.sql` completo (ver acima), e aplicar manualmente no SQL Editor do Supabase. Nota: a flag correta hoje é `--from-schema`/`--to-schema` (a CLI removeu `--from-schema-datamodel`/`--to-schema-datamodel` numa versão mais nova do Prisma).
- **RLS (Row Level Security)**: `supabase/policies.sql` é a fonte da verdade das políticas de RLS para as 13 tabelas — todas com padrão confirmado em produção (inspecionado, não suposição), nenhuma pendente. Padrão: policy `FOR ALL TO public`, nome `{tabela}_por_loja` (exceto `ordens_servico` → `ordens_por_loja`), `USING ("lojaId" IN (SELECT usuarios."lojaId" FROM usuarios WHERE usuarios."authUserId" = auth.uid()::text))` — subquery direta na tabela `usuarios`, sem função auxiliar, com cast explícito de `auth.uid()` pra `::text`. Confirmado pra `clientes_por_loja`, `veiculos_por_loja` e `ordens_por_loja`. Há **duas exceções estruturais** a esse padrão, ambas confirmadas em produção:
  - `lojas`: a policy `lojas_por_usuario` compara `id` — a própria linha, já que `lojas` não tem coluna `lojaId` — com o `lojaId` do usuário logado, em vez de `lojaId = lojaId` como nas demais.
  - `usuarios`: a policy `usuarios_por_loja` **não usa subquery** — compara `"authUserId" = auth.uid()::text` direto. Uma subquery que lê da própria `usuarios` dentro da policy de `usuarios` reavalia a RLS de `usuarios` de novo, causando recursão infinita (Postgres `42P17` — "infinite recursion detected in policy for relation usuarios"). `supabase/policies.sql` teve esse bug até 2026-07-01 (replicava o padrão de subquery aqui também), causando HTTP 500 em todo login ao consultar `usuarios` — descoberto rodando a query contra o projeto Supabase DEV (`gghbrjaeuzvezlzjabiy`) e corrigido comparando com a policy real de produção.

  Igual às migrations, **aplicar esse arquivo não é automático** — sempre que uma policy mudar (aqui ou direto no painel do Supabase), atualize os dois lados manualmente para não ficarem dessincronizados.
- Docker/Express/Firebase foram removidos do projeto e **não devem ser reintroduzidos**.

## Como adicionar um novo usuário/funcionário
Criar um usuário no Supabase Auth **não** cria automaticamente a linha correspondente na tabela `usuarios` — sem essa linha, `useAuth.ts` nunca encontra o `usuario` (fica `null`) e o app mostra a tela `ContaNaoVinculada` em vez de liberar acesso, mesmo com login válido. Até esse vínculo ser automatizado, o processo é manual, sempre nesta ordem:

1. **Criar o usuário no Supabase Auth**: painel do Supabase → Authentication → Users → "Add user". Preencher email e senha (ou enviar convite, conforme o caso) e confirmar a conta.
2. **Copiar o `id` gerado** para esse usuário na tabela `auth.users` (é o UUID mostrado na lista de Users).
3. **Inserir manualmente a linha correspondente na tabela `usuarios`** (SQL Editor do Supabase), com:
   - `authUserId` = o `id` copiado no passo 2 (mesmo formato usado em `usuarios.authUserId` e comparado contra `auth.uid()::text` nas policies de RLS — ver `supabase/policies.sql`);
   - `lojaId` = o `id` da loja correta em `lojas`, para esse funcionário ficar no tenant certo;
   - demais campos (`nome`, `email`, `role`) preenchidos conforme o funcionário.

Sem os passos 2–3, o login funciona (a sessão do Supabase Auth é criada normalmente) mas o app trata a conta como não vinculada a nenhuma loja.

**Atenção ao colar UUIDs no Table Editor**: colar um UUID copiado (`authUserId`, `lojaId`) direto nos campos do formulário visual do Table Editor às vezes gruda uma quebra de linha invisível no final do valor — o erro de foreign key que o Postgres retorna nesse caso não deixa óbvio que a causa é essa. Se um INSERT/UPDATE falhar por FK mesmo com o valor parecendo correto visualmente, rode `SELECT id, length(id) FROM tabela WHERE ...;` — um UUID válido sempre tem `length = 36`; se vier diferente, limpe com `UPDATE tabela SET id = regexp_replace(id, '[^a-zA-Z0-9\-]', '', 'g') WHERE ...;`. Pra evitar o problema, prefira inserir via SQL Editor (`INSERT INTO ...`) em vez do formulário visual quando for copiar/colar UUIDs.

## Perfil local = loja do usuário logado (não existe mais SelecionarPerfil)
Até meados de 2026 o app tinha uma tela `SelecionarPerfil` (pré-login, cada "perfil" era um id aleatório gerado no navegador) porque não havia autenticação. Com o Supabase Auth em produção, cada usuário logado já tem um `usuario.lojaId` real — manter os dois sistemas em paralelo (perfil local solto + loja de verdade) desconectava os dados de negócio do dono real da loja. Isso foi corrigido:

- **`SelecionarPerfil.tsx`/`useSelecionarPerfil.ts` foram removidos.** `src/App.tsx` não guarda mais `perfilAtivo` em `useState` — em vez disso, `ProtectedRoute` (que já resolve `usuario` via `useAuth`) passa o `usuario` para os filhos via render-prop (`children: (usuario: Usuario) => ReactNode`), e `App.tsx` usa `usuario.lojaId` diretamente.
- **O namespace do `usePersistedState` não mudou de formato**: continua lendo `sessionStorage['wrapos_perfil_ativo']` para montar `wrapos_perfil_<id>_<entidade>` (zero mudança em `AppContext.tsx`). Só passou a ser preenchido com `usuario.lojaId` (gravado de forma síncrona em `App.tsx`, antes do `AppProvider` montar) em vez de um id aleatório escolhido na tela antiga.
- **Login vai direto pra `Início`**, sem tela de seleção — a única exceção é a migração de dados legados abaixo.
- **Migração de dados de um perfil local antigo**: se o navegador ainda tem perfis salvos da era pré-login (`localStorage['wrapos_perfis']`), `src/hooks/useMigrarPerfilAntigo.ts` + `src/pages/MigrarPerfilAntigo.tsx` oferecem, **uma única vez por loja**, vincular esses dados ao `usuario.lojaId` atual (copia as chaves `wrapos_perfil_<idAntigo>_*` para `wrapos_perfil_<lojaId>_*`, sem apagar o original). A flag de "já resolvido" é `localStorage['wrapos_migracao_feita_<lojaId>']` — **escopada por lojaId, não global** — para funcionar corretamente se o mesmo navegador algum dia logar com outra loja.
- **e2e**: como não há mais formulário de criação de perfil, `e2e/helpers.ts` não tem mais `criarPerfil()` — `abrirApp(page)` só faz `goto('/')` e espera a Home. O dataset de demonstração usado nos specs (15 OS, 3 instaladores, 8 serviços, 6 produtos com 2 críticos, 1 meta mensal) continua vindo do valor inicial (`initialOrdens` etc. em `AppContext.tsx`) que `usePersistedState` usa como fallback quando não há chave no localStorage para aquele `lojaId` — cada teste roda num `BrowserContext` isolado, então isso equivale ao antigo `comExemplo=true`. **Exceção: `clientes` e `veiculos`** não seguem mais esse fallback desde a migração pra Supabase (ver "Migração de entidades pro Supabase" abaixo) — precisam de linhas reais nas tabelas `clientes`/`veiculos` do projeto Supabase usado pelos testes.

## Migração de entidades pro Supabase (localStorage → nuvem, incremental)
A partir de 2026-07-01, WrapOS migra a persistência de localStorage pro Supabase **entidade por entidade**, começando pela mais simples: `clientes`, seguida por `veiculos` em 2026-07-06 (segunda entidade). O padrão abaixo é o modelo a seguir para as próximas 8 (`ordens`, `agendamentos`, `instaladores`, `lancamentos`, `produtos`, `garantias`, `meta`, `configuracoes`, `servicos`).

- **⚠️ Pré-requisito obrigatório antes de migrar qualquer entidade: chave primária composta `(lojaId, id)` no schema.** Todo `initial*` em `AppContext.tsx` usa ids curtos e fixos (`c1`, `v1`, `s1`, `i1`...) — únicos só *dentro* do namespace local de cada loja no localStorage. `clientes.id` era `@id` sozinho (PK global) até 2026-07-02, e isso quebrou de verdade: duas lojas diferentes que nunca editaram o dado de exemplo, ambas tentando migrar seus clientes locais pro Supabase, colidiam em `clientes_pkey` (a segunda a migrar um "c1" era rejeitada, `23505 duplicate key`) — descoberto rodando o seed de teste (`e2e/global-setup.ts`) contra o projeto DEV, onde a loja de teste e a loja pessoal de outro usuário já tinham ambas os ids clássicos `c1..c10`. Corrigido trocando `Cliente.id @id` por `@@id([lojaId, id])`, e as FKs de `Veiculo.cliente`/`OrdemServico.cliente` para compostas `(lojaId, clienteId) → Cliente(lojaId, id)` (não dá pra ter FK simples apontando pra PK composta) — ver `supabase/migrations/003_clientes_chave_composta.sql`. **O mesmo tratamento foi replicado pra `Veiculo` em 2026-07-06**, antes de migrar `veiculos`: `Veiculo.id @id` virou `@@id([lojaId, id])`, e a FK de `OrdemServico.veiculo` virou composta `(lojaId, veiculoId) → Veiculo(lojaId, id)` — ver `supabase/migrations/004_veiculos_chave_composta.sql`. **Toda próxima entidade migrada precisa do mesmo tratamento antes do primeiro deploy real** (não só em teste): se dois `initial*` de duas lojas diferentes nunca editados colidem por id, a migração vai falhar do mesmo jeito.
- **`src/hooks/useClientesSupabase.ts`** e **`src/hooks/useVeiculosSupabase.ts`** substituem `usePersistedState('clientes'/'veiculos', ...)` só para essas duas entidades, ambos seguindo exatamente o mesmo formato (mesma função `paraLinha`/normalização, mesma flag de migração one-time, mesma mutação otimista — ver abaixo). Recebem `lojaId` (lido em `AppContext.tsx` de `sessionStorage['wrapos_perfil_ativo']` — o mesmo valor que `usePersistedState` já usava, e que desde a migração de perfil é sempre `usuario.lojaId`; `AppContext.tsx` continua sem importar `useAuth` diretamente).
- **Ao montar**: busca `supabase.from('clientes'|'veiculos').select('*').eq('lojaId', lojaId)`. RLS (`clientes_por_loja`/`veiculos_por_loja`) já restringe a mesma coisa por conta própria — o filtro explícito é defesa em profundidade, não a única barreira.
- **Migração one-time de dados legados**: antes da busca definitiva, cada hook verifica se há linhas ainda só em `localStorage['wrapos_perfil_<lojaId>_clientes'|'_veiculos']` que não existem no Supabase (compara por `id`) e faz `insert` das que faltam. Flag de "já migrei" é `localStorage['wrapos_clientes_migrados_<lojaId>']`/`['wrapos_veiculos_migrados_<lojaId>']` — **escopada por lojaId**, mesmo padrão da flag `wrapos_migracao_feita_<lojaId>` descrita em "Perfil local = loja do usuário logado" acima. Não apaga a chave antiga do localStorage (só para de lê-la depois de migrar).
- **Mutação otimista**: `adicionarCliente`/`editarCliente`/`removerCliente` e `adicionarVeiculo`/`editarVeiculo`/`deletarVeiculo` (chamados internamente pelas funções de mesmo nome em `AppContextType`) atualizam o estado local primeiro, disparam a chamada Supabase depois, e revertem o estado local + `toast.error` (Sonner) se a chamada falhar. Ambos os `adicionar*` geram o `id` no cliente (mesmo `uid()` de sempre) e retornam sincronamente — necessário porque chamadores existentes (ex: `useClientes.ts`, `useAgendamento.ts`, `useOrdemServico.ts`, `CheckinRapido.tsx`) usam o id retornado na mesma função pra criar um veículo/OS/agendamento vinculado, sem esperar round-trip de rede.
- **Detalhe de tipo (só em `clientes`)**: a coluna `dataCadastro` no Postgres é `TIMESTAMP`, mas o tipo `Cliente.dataCadastro` no frontend é uma string `'YYYY-MM-DD'` pura (usada em comparações tipo `.startsWith(mesAtual)` em `useEquipe.ts`/`useMetas.ts`). `useClientesSupabase.ts` normaliza a data de volta pra `'YYYY-MM-DD'` (`.slice(0, 10)`) ao ler do Supabase — sem isso, essas comparações quebram silenciosamente. `Veiculo` não tem esse problema (nenhum campo de data — `useVeiculosSupabase.ts` não precisa de normalização equivalente).
- **Gap conhecido, não corrigido ainda**: `src/utils/backup.ts` (export/import manual) só enxerga `localStorage`, então não inclui mais `clientes` nem `veiculos` no backup a partir desta migração — vai piorar a cada entidade migrada até o backup em si ser repensado (fora do escopo desta mudança).
- **Status**: `003_clientes_chave_composta.sql` foi gerado e `schema.sql` regenerado (2026-07-02), mas **ainda não aplicado em nenhum banco** (nem DEV, nem produção) — aplicar continua manual (ver "Para alterar o banco" acima). Até isso rodar no SQL Editor, qualquer segunda loja real que tentar migrar clientes locais intocados vai bater em `23505 duplicate key` em produção (na prática o seed de teste, `e2e/global-setup.ts`, não expõe esse bug porque só há uma loja com `clientes` reais no projeto DEV hoje). `004_veiculos_chave_composta.sql` foi gerado, `schema.sql` regenerado, e **já aplicado no projeto Supabase DEV** (`gghbrjaeuzvezlzjabiy`) em 2026-07-06 — aplicar em produção continua pendente, mesma ressalva de 003.
- **Impacto no e2e**: o dataset de demonstração (`initialClientes`/`initialVeiculos` em `AppContext.tsx`) não é mais usado como fallback pra essas duas entidades — um `lojaId` novo, sem linhas em `clientes`/`veiculos` no Supabase, simplesmente começa com 0 (correto para uma loja real nova; quebra a suposição dos specs que dependiam do seed automático de 10 clientes/veículos). Ajustes feitos na suíte por causa disso:
  - **Seed/cleanup em `e2e/global-setup.ts`** (função `semearDados`, antes `semearClientes`): antes de qualquer spec rodar, loga com a conta e2e via `@supabase/supabase-js` (fora do browser), resolve o `lojaId` da loja de teste, apaga tudo em `veiculos` e depois em `clientes` dessa loja (nessa ordem — FK composta de `veiculos.clienteId`), reinsere os mesmos 10 clientes de `initialClientes` e, na sequência, os mesmos 10 veículos de `initialVeiculos` (ambos mantidos como cópia local em `global-setup.ts` — não importa `AppContext.tsx` direto num script Node puro pra não puxar a árvore de imports do app, que assume DOM/React). Sem isso, cada rodada herdaria lixo (linhas criadas/editadas) da rodada anterior, já que o Supabase é estado real, não mais localStorage isolado por `BrowserContext`.
  - **Suíte agora roda serial** (`fullyParallel: false`, `workers: 1` em `playwright.config.ts`) — **não é regressão, é intencional**. Antes, paralelismo (`fullyParallel: true`) era seguro porque cada teste tinha seu próprio localStorage isolado por `BrowserContext`; nada vazava entre specs concorrentes. Agora que `clientes`/`veiculos` são tabelas compartilhadas na mesma loja de teste, specs paralelos que leem/escrevem a mesma linha podem ter *lost update* (dois workers leem o mesmo `totalGasto`, cada um escreve sua própria conta, um dos dois updates se perde). Auditoria feita em 2026-07-02 nos 6 specs atuais não achou nenhuma escrita concorrente real na mesma linha de `clientes` (`01` e `03` usam o mesmo cliente "Gabriela Alves" mas só `01` escreve `totalGasto`; `_debug-repro` inativa "Felipe Santos" mas nenhum outro spec depende dele); `04-exclusao-cliente.spec.ts` cria um veículo novo ("Fiat Uno") mas nenhum outro spec lê/escreve essa mesma linha. Ainda assim, esse acoplamento é coincidência frágil, não garantia estrutural, e vai piorar de forma concreta (não mais coincidência) quando `ordens` migrar: o contador sequencial `numero` (`Math.max(...) + 1`) tem uma condição de corrida real entre workers concorrentes lendo o mesmo estado antes de escrever. Reavaliar workers>1 só se/quando os specs passarem a usar dados isolados por teste na nuvem (não uma loja de teste compartilhada).

## Backup manual (rede de segurança até a migração pro Supabase)
Enquanto todos os dados de negócio (clientes, ordens, agendamentos, financeiro, estoque etc.) vivem só no `localStorage` do navegador, sem backup automático, `src/utils/backup.ts` oferece:
- `exportarBackup()` — varre todas as chaves do `localStorage` que começam com `wrapos_` (inclui as por perfil, `wrapos_perfil_<id>_<entidade>`) e baixa um JSON `wrapos-backup-{data}.json`.
- `importarBackup(arquivo: File)` — lê um JSON nesse mesmo formato e restaura as chaves no `localStorage`.
UI em Configurações (`src/pages/Configuracoes.tsx` + `src/hooks/useConfiguracoes.ts`), seção "Backup", com confirmação explícita em modal antes de sobrescrever dados na importação. Após importar, a página recarrega (`window.location.reload()`) porque `usePersistedState` só lê do `localStorage` na montagem inicial.

## Design System — atenção: dois sistemas coexistem
1. `src/index.css` define `--surface-900/800/700`, `--ui-border`, `--ui-text` — **esse é o que está plugado no tailwind.config** (classes `bg-surface-700`, `text-ui-text`, `border-ui-border`). Usar esse pra qualquer componente/página nova.
2. `src/styles/tokens.css` define `--wrap-bg`, `--wrap-surface`, `--wrap-text`, `--wrap-accent` etc — sistema mais antigo, ainda usado em ~199 lugares no código existente, importado em `main.tsx`. Não remover sem revisar todo uso primeiro.
Fontes: Syne (títulos, `font-display`) + DM Sans (corpo, `font-sans`).
Cor de acento (`--wrap-accent-rgb`) é injetada em runtime via ThemeContext.
